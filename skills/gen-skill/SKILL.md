# Gen Skill：从对话中提取并生成 Skill

把"刚刚做过一遍的事"沉淀成可被未来复用的 Cursor Skill。本 skill 分两步：先判断当前对话是否值得生成，再生成 `SKILL.md` 并写入正确位置。

---

## 触发场景

- 用户输入 `/gen-skill` 或类似指令
- 用户说"把刚才的流程保存成 skill / 技能 / 命令"
- 用户说"提取 skill"、"生成 skill"、"沉淀一下这次的工作"
- 一段较完整的工作流程结束后，用户希望复用它

## 执行流程

### 步骤 1：回顾对话，判断是否应该生成

回顾本次对话中实际完成的事情（不要只看最后一条消息），从以下维度判断：

**适合生成的信号（满足任意 2 条）：**
- 完成了一个有明确开始和结束的工作流程（实现功能、修复 bug、重构、调研、配置环境等）
- 流程具备通用性，未来很可能再做一遍（哪怕换个项目/模块）
- 包含足够上下文：步骤顺序、关键决策、踩过的坑、所需工具
- 用户明确表达过"以后还会用到"或"这是一个固定流程"

**不适合生成的信号（命中任意 1 条即拒绝）：**
- 仅闲聊、问答、概念解释
- 仅做了微小修改（改变量名、加一行日志、调一个常量）
- 一次性的、强项目耦合的临时操作（如"删掉这个特定文件"）
- 流程不完整，缺少关键步骤或上下文

**输出判断结果**（先告诉用户）：

```
判断：是否生成 skill = <是 / 否>
原因：<一句话说明>
```

如果判断为"否"，停止；可以建议用户继续完成流程后再调用，或描述更多细节。

### 步骤 2：与用户确认 skill 关键信息

判断为"是"后，先用一段简短文字向用户提案：

```
建议生成 skill：
- name: <kebab-case-name>
- description: <一句话功能 + 触发时机>
- 核心步骤: <3-5 条要点预览>
- 存放位置: <个人 ~/.cursor/skills/ 或 项目 .cursor/skills/>
```

如有 `AskQuestion` 工具，对存放位置等关键选项使用结构化询问；否则在文本里直接问，并等待用户确认或调整。如果用户已经在原始请求里指定了名字/位置，跳过询问，直接采用。

### 步骤 3：生成并写入 SKILL.md

#### 3.1 命名与位置

- `name`：小写英文 + 连字符，≤ 64 字符，含义清晰（如 `fix-flaky-test`、`migrate-mysql-schema`）
  - 避免 `helper`、`utils`、`tools` 这类模糊名
- 路径：
  - 个人：`~/.cursor/skills/<name>/SKILL.md`
  - 项目：`<repo>/.cursor/skills/<name>/SKILL.md`
- 严禁写入 `~/.cursor/skills-cursor/`（系统保留目录）
- 创建目录使用 `mkdir -p`，文件用 Write 工具

#### 3.2 SKILL.md 模板

严格遵守 YAML frontmatter + Markdown 正文：

```markdown
---
name: <kebab-case-name>
description: <第三人称、一句话说明 WHAT + WHEN，包含明显触发词>
---

# <Skill 标题>

<一句话点题：这个 skill 解决什么问题。>

## 触发场景
- <场景 1>
- <场景 2>

## 执行流程

### 步骤 1：<动词开头>
<具体做什么、用什么工具、注意点>

### 步骤 2：<动词开头>
...

## 关键规则 / 注意事项
- <从对话中提炼的踩坑点、约束、最佳实践>

## 示例（可选）
<最小可运行的输入/输出示例>
```

#### 3.3 写作要求

- **第三人称、祈使句**："读取 X"，不是"我会读取 X"或"你可以读取 X"
- **description 同时包含 WHAT 和 WHEN**，并放入用户可能说的触发词（命令名、动词、领域名词）
- **正文 ≤ 200 行**；超出请用 `reference.md` / `examples.md` 渐进披露，链接保持一层深度
- **优先复用对话中已用过的工具名、命令、文件路径**，让 skill 与真实工作流一致
- **明确"做什么"和"不做什么"**，避免给模型过多自由度而偏离正确流程
- **去掉一次性细节**：把对话里特定的项目名、临时文件名抽象成占位符 `<...>`
- **不要包含时效性内容**（如"在 2026 年 4 月之前用旧 API"）

### 步骤 4：写入并向用户报告

写入文件后，向用户输出：

```
已生成 skill：<绝对路径>
- name: <name>
- description: <description>
- 行数: <N>
下一步可以：在新对话中触发该 skill 验证效果，或继续编辑补充示例。
```

如适用，附上文件引用块（CODE REFERENCE 格式）让用户可以一键打开。

---

## 反模式

- ❌ 把整段对话原文塞进 SKILL.md（应提炼流程，不是复制聊天记录）
- ❌ description 写成"帮你做 X"或"我能做 X"（用第三人称）
- ❌ 跳过判断步骤直接生成（容易产出垃圾 skill 污染技能库）
- ❌ 在 `skills-cursor/` 下创建文件
- ❌ 同时给出多个并列方案让模型选（应给一个默认 + 必要时的逃生口）
- ❌ name 与已有 skill 重名而不提醒用户

## 验收清单

生成完毕前自检：

- [ ] frontmatter 的 `name` 和 `description` 合法且具体
- [ ] description 含触发词，第三人称
- [ ] 正文 ≤ 200 行，术语统一
- [ ] 流程步骤可独立执行，不依赖本次对话的具体上下文
- [ ] 路径正确（个人 vs 项目），未写入 `skills-cursor/`
- [ ] 已向用户报告生成结果与路径
